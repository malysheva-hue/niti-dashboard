# Источники данных дашборда

Зафиксировано в коде (v1.4). Менять только согласованно — все 3 источника
имеют разную свежесть и разную семантику.

## Иерархия источников

| Источник | Папка / маска | Что даёт | Свежесть | Используется для |
|---|---|---|---|---|
| **Воронка ВБ** | `inputs/*воронка*.xlsx` | Выручка, заказы, выкупы, конверсии, показы, средняя цена, % выкупа, ВБ-клуб | **Ежедневно**, источник правды для текущего дня | Блок «За {вчера}», карточки дня, дневная динамика |
| **Шаблон цен** | `inputs/Шаблон*цен*.xlsx` | Текущая цена, СПП %, скидка, розничная / акционная цена | По запросу РОПа (не ежедневно) | Контроль цен, OOS-расчёт остатков |
| **План-Факт MPSTATS** | `inputs/План_факт*.xlsx` | Маржа, прибыль, ДРР, ROMI, CPC, остатки, оборачиваемость, заморозка | **Отстаёт на 1-2 дня** | Экономика, история, аномалии правил |
| **CSV-агрегаты** | `data/csv-year/` | Помесячные / годовые агрегаты, дневные ряды | История (июнь 2025 — текущий месяц) | 30-дневная история SKU, MoM-сравнения |

## Правила использования

### Текущий день
Все метрики дня (выручка, заказы, шт, выкупы, конверсии) — только из **Воронки**.
`Дневная_*.csv` для этих целей не используется (отстаёт по факту выгрузки и
повторяет данные MPSTATS).

### Прибыль и маржа дня
Прибыль воронка не содержит. Рассчитывается как:

```
profit_day = Σ (orders_sku_funnel × profit_per_unit_sku_mpstats)
margin_day = profit_day / revenue_day × 100
```

`profit_per_unit_sku` берётся из «План-Факт» (`Прибыль на 1 выкуп, руб`).
Если для SKU нет данных в MPSTATS или MPSTATS отстаёт > 1 дня — поле
помечается `is_stale=true` и в UI вместо числа показывается «ждёт загрузки».

### Сравнение «vs вчера»
Только между двумя соседними воронками. Если предыдущей воронки нет —
показывается прочерк, проценты не вычисляются.

## Staleness (защита от отстающих данных)

В каждом блоке метрик из MPSTATS добавлены поля:

```json
{
  "metric": "margin_pct",
  "value": 23.5,
  "as_of_date": "2026-05-18",
  "is_stale": false,
  "source": "mpstats"
}
```

Глобальный `mpstats_meta` в `data.json` и `ai_summary.json`:

```json
{
  "last_known_date": "2026-05-19",
  "today": "2026-05-20",
  "days_behind": 1,
  "is_stale": false
}
```

### Правило `is_stale`
- `is_stale = (today - last_known_date) > 1 день`
- Если `is_stale=true` — на главной странице вверху появляется жёлтая
  плашка: *«⏳ Часть данных за {date} ещё не загружена из MPSTATS»*.
- В блоке «За {вчера}» цифры из MPSTATS отображаются серым курсивом со
  словами «ждёт загрузки».

### Аномалии
Эти случаи помечаются `is_stale=true` для конкретного SKU и **исключаются
из триггеров правил движка**:
- Выручка/прибыль/заказы за день < 10% от 7-дневного среднего того же
  показателя → значит, данные не доехали.
- Маржа = 0 при ненулевых заказах → MPSTATS не записал значение.

## Что куда не идёт

- **Никогда** не показываем «сегодняшние» цифры из MPSTATS, если ему >48 часов.
- **Никогда** не парсим воронку для маржи/прибыли — она их не содержит.
- **Никогда** не используем `Дневная_*.csv` для основных метрик дня — только
  как fallback-резерв (на момент v1.4 вообще не используется).

## Что РОПу нужно знать для добавления нового источника

1. Понять, к какой категории относится: дневной фактический (как воронка),
   справочный (как цены), экономический (как MPSTATS) или историчный (как CSV).
2. Дописать парсер в `build.py` рядом с `parse_*` соответствующего типа.
3. Указать `as_of_date` и `is_stale` для каждой метрики если данные могут отставать.
4. Обновить эту таблицу.
5. Если новый источник заменяет существующий — обновить engine/operators
   (если оператор читает конкретное поле) и YAML-правила.
